Economic indicators are drawn from external APIs where possible, passed through a selection layer that prefers live API values, and then fed into the TS-Transformer to produce inflation forecasts, deflation probabilities, confidence scores, and explanatory outputs.

## 4.2 System Implementation

The developed system, referred to here as the *Inflation and Deflation Prediction System*, follows a three-tier architecture consisting of a frontend presentation layer, a backend application layer, and a persistence layer. A separate machine learning module handles model training and inference. These parts were not developed in isolation; they were wired together so that a forecast generated by the model can be stored, explained, displayed on a dashboard, exported as a report, and delivered to the user through notifications.

### 4.2.1 Overall System Flow

When a user requests a prediction for a selected country, the frontend sends an authenticated request to the FastAPI backend. The backend first resolves the country’s economic indicators through an API-first selection service. This service refreshes stale data from connected APIs when necessary, evaluates candidate values from sources such as FRED, IMF, World Bank, Trading Economics, and Exchange Rate API, and selects the most appropriate value for each indicator. Manual or locally stored values are used only when no API provides the required field.

After the input vector is assembled, the backend invokes the TS-Transformer inference engine. The engine constructs a sequence of 24 historical time steps, enriches the input with event and sentiment features, normalises the data using the trained scaler, and runs the model to obtain forecasts for multiple horizons. The prediction result, together with explainability metadata, is stored in PostgreSQL and returned to the frontend for display. The same workflow supports report generation and notification delivery.

### 4.2.2 Frontend Implementation

The frontend was implemented with Next.js and TypeScript. Next.js was chosen because it supports server-side rendering, route-based page organisation, and efficient development of multi-role applications. TypeScript was used to enforce type safety across components, API clients, and shared data structures, which reduced runtime errors during integration with the backend.

The interface is divided according to user role. Regular users access the User Dashboard, where they can view country summaries, run predictions, inspect forecast charts, download reports, and read economic news. Analysts access a separate workspace with deeper analytics, scenario testing, explainability tools, backtesting views, and research-oriented pages. Administrators manage users, API credentials, model training jobs, economic events, and platform settings.

Client-side data fetching is handled through a central API service built on Axios. State for authentication, active country selection, and notifications is managed with lightweight client stores. Charts are rendered with Recharts, while page transitions and dashboard panels use Framer Motion for a polished but restrained user experience. The visual design follows a monochrome design system so that attention remains on the economic data rather than decorative styling.

### 4.2.3 Backend Implementation

The backend was implemented with FastAPI. FastAPI provides asynchronous request handling, automatic OpenAPI documentation, and strong integration with Pydantic for request validation. This was important because the system exposes many endpoints for predictions, intelligence features, administration, and external API management.

The backend is organised into routers, services, models, and schemas. Routers define the HTTP endpoints. Services contain the business logic, including prediction execution, dashboard aggregation, intelligence analysis, report generation, and API synchronisation. SQLAlchemy models represent database tables, while Pydantic schemas define the structure of requests and responses.

A major backend component is the prediction service, which calls the indicator selection service, prepares model input, invokes the TS-Transformer engine, and persists the resulting forecast. Another important component is the intelligence platform service, which supports the Intelligence Hub, data lineage, reliability scoring, forecast archives, early warnings, and natural-language query features. Background schedulers keep external API data current by periodically syncing FRED, IMF, World Bank, Trading Economics, exchange rates, news, and related sources.

### 4.2.4 Database Implementation

PostgreSQL was used as the primary database. It stores user accounts, roles, predictions, economic data, API configuration, exchange rate history, intelligence artefacts, training jobs, and notification records. The database was accessed asynchronously through SQLAlchemy with `asyncpg`, which allows the backend to handle multiple concurrent prediction and dashboard requests without blocking.

Database migrations were managed with Alembic. This made it possible to evolve the schema safely as new intelligence features were added, including tables for forecast archives, backtest sessions, model experiments, and prediction metadata such as reliability scores and data lineage.

### 4.2.5 Machine Learning Implementation

The machine learning component was implemented in PyTorch. The core model is the TS-Transformer, a time-series transformer designed for multivariate macroeconomic forecasting. The model accepts a sequence of macroeconomic features per country, passes them through an input projection layer, adds positional encoding, and processes the sequence using a stack of transformer encoder layers with multi-head self-attention.

The model produces several outputs simultaneously:

- predicted inflation rates across multiple horizons;
- deflation probabilities;
- trend direction;
- confidence scores;
- risk levels.

Training and inference share the same preprocessing pipeline. During training, economic CSV datasets for countries such as Nigeria, the United States, the United Kingdom, Ghana, and South Africa are loaded, cleaned, normalised, and converted into sliding-window sequences. During inference, live or API-selected indicators are mapped into the same feature space used at training time. If a trained checkpoint is available, the system loads `best_model.pt` together with the saved scaler; otherwise, a controlled heuristic fallback is used to avoid service failure.

### 4.2.6 TS-Transformer with Attention Mechanism

The attention mechanism is implemented through multi-head self-attention in each encoder layer. For a sequence of 24 time steps, the model learns which past periods deserve greater weight when forecasting future inflation. This is particularly useful in macroeconomic data, where distant shocks such as oil price spikes, exchange rate movements, or policy changes may still influence current inflation dynamics.

At inference time, attention weights from the first encoder layer are extracted and converted into an explainability heatmap. These weights are also used to rank feature importance, allowing the system to indicate whether CPI, interest rate, exchange rate, unemployment, or event-related signals contributed most strongly to a given forecast. This design connects the model’s internal behaviour to user-facing explanation rather than presenting predictions as opaque numerical outputs.

### 4.2.7 API Integrations

To keep the system current without daily manual updates, several external APIs were integrated:

- **FRED API** for United States macroeconomic series such as CPI, core inflation, interest rates, and labour market indicators;
- **IMF API** for cross-country macroeconomic outlook data;
- **World Bank API** for development and economic indicators;
- **Trading Economics API** for supplementary market and macro data;
- **Exchange Rate API** for live currency conversion rates.

Each API has an admin configuration page where credentials, refresh intervals, and sync status can be monitored. When a prediction or dashboard request is made, the system may refresh stale API data for the selected country before resolving the final indicator values. This API-first strategy was adopted so that the platform remains aligned with official and market data releases.

### 4.2.8 Authentication System

Authentication was implemented using JWT-based access and refresh tokens. Users register with email and password, after which credentials are hashed with bcrypt before storage. Login returns short-lived access tokens and longer-lived refresh tokens. Protected routes verify the user’s role before granting access to user, analyst, or admin functionality.

Google OAuth was also integrated on the frontend for alternative sign-in. Role-based access control ensures that ordinary users cannot access admin API settings, while analysts can view advanced model and backtesting tools not exposed on the standard user dashboard.

### 4.2.9 Report Generation Module

The report generation module produces downloadable economic intelligence documents. When a user requests a report, the backend compiles the latest forecast, confidence bands, influencing factors, and country context into a structured PDF using ReportLab. Reports can also be exported in spreadsheet form where tabular analysis is more appropriate. This module is especially useful for users who need offline copies of forecasts for meetings, academic review, or policy discussions.

### 4.2.10 Notification System

The notification system alerts users when important events occur, such as the completion of a prediction, availability of a new report, model retraining results, or significant economic warnings. Notifications are stored in the database and displayed inside the dashboard. Email delivery is supported through SMTP or Resend, depending on deployment configuration. This ensures that users do not have to remain on the platform continuously to know when fresh forecasts are ready.

### 4.2.11 User Dashboard

The User Dashboard serves as the main workspace for non-technical users. It presents economic summary cards, a country focus bar, tracked countries, inflation forecasts, exchange rate details, and quick links to predictions, analytics, reports, and news. The dashboard was designed to give an immediate overview of economic conditions without requiring the user to interpret raw model internals.

### 4.2.12 Analyst Dashboard

The Analyst Dashboard extends the standard user experience with tools intended for deeper investigation. Analysts can inspect model versions, review data source coverage, explore scenario simulations, study explainability outputs, compare accuracy across countries, and use the Intelligence Hub for narrative insights, early warnings, and backtesting. This separation prevents casual users from being overwhelmed while still giving technical users the depth they need.

### 4.2.13 Admin Dashboard

The Admin Dashboard provides full platform control. Administrators can manage users, configure external APIs, monitor sync health, upload or review economic data, schedule model training, manage economic events, edit site content, and inspect platform analytics. The admin interface is critical for maintaining data quality and ensuring that the prediction system remains operational as API credentials, model versions, and user demand change over time.

## 4.6 Model Training and Implementation

### 4.6.1 Data Collection

Training data was obtained from structured macroeconomic datasets and supplemented by API-sourced indicators. Country-specific CSV files were prepared for Nigeria, the United States, the United Kingdom, Ghana, and South Africa. These files contained fields such as CPI, GDP growth, interest rate, exchange rate, oil price, government spending, employment rate, money supply, and inflation rate. Additional live values were integrated from FRED and other APIs during system operation.

### 4.6.2 Data Preprocessing

The preprocessing stage involved several steps. First, the raw files were loaded and sorted chronologically. Column names were standardised so that training data and live API data used the same feature names. Missing values were handled through forward filling and sensible defaults where necessary. For example, employment rate could be derived from unemployment rate when only the latter was available.

### 4.6.3 Feature Engineering and Sequence Generation

The model does not consume a single snapshot of indicators. Instead, it uses a sliding window of 24 time steps to preserve temporal structure. Lagged inflation was included as an additional feature because past inflation is one of the strongest predictors of future inflation. Event features were also appended to represent the count, severity, impact, and recency of economic events.

### 4.6.4 Normalisation

All numeric features were scaled using `MinMaxScaler` so that variables with large magnitudes, such as money supply or exchange rate, did not dominate variables with smaller ranges, such as GDP growth. A separate scaler was used for the target inflation series so that evaluation could be performed in the original percentage scale.

### 4.6.5 Training and Validation Process

The dataset was divided into training, validation, and test subsets using a time-aware split so that future data were not leaked into training. The TS-Transformer was trained for up to 80 epochs with a batch size of 32, learning rate of 0.0002, and early stopping patience of 12 epochs. A cosine learning rate schedule was applied, and gradient clipping was used to improve stability.

The model configuration used during implementation was as follows:

- model dimension: 128;
- attention heads: 8;
- encoder layers: 4;
- feedforward dimension: 512;
- dropout: 0.1.

Training optimised a multi-task loss covering inflation regression, deflation probability, trend classification, confidence estimation, and risk scoring. The best checkpoint was saved as `best_model.pt` and loaded by the inference engine at runtime.

### 4.6.6 Role of the Attention Mechanism

The attention mechanism allows the model to focus on the most informative time steps in the input sequence. In conventional recurrent models, distant information may fade gradually. In contrast, self-attention computes direct relationships between all time steps in the window. For inflation forecasting, this means the model can associate a current inflation movement with an earlier exchange rate shock or policy-related event even if many months separate them.

During evaluation, attention weights were extracted and mapped to user-readable explanations. This improved trust in the model because forecasts were no longer presented without justification.

## 4.8 Discussion of Results

The results show that the proposed TS-Transformer with attention mechanism can be implemented successfully as part of a full web-based inflation forecasting system. The model did not merely exist as a training script; it was integrated into a production-style workflow involving API-driven data ingestion, database storage, role-based dashboards, explainability, and report generation.

### 4.8.1 Prediction Accuracy

An overall accuracy of 80.1% is a strong outcome for inflation forecasting, especially considering the volatility of macroeconomic series in emerging economies. Nigeria’s result of 79.2% on 57 test records is particularly important because Nigeria was the most data-rich country in the evaluation set. South Africa achieved the lowest MAPE at 7.95%, although this was based on only five test records and should therefore be interpreted cautiously.

### 4.8.2 Strengths of the Model

The main strengths observed during implementation were:

1. **Temporal modelling capability** — the transformer encoder captured patterns across 24-month input windows more effectively than a simple feedforward baseline;
2. **Multi-output prediction** — inflation, deflation probability, trend, confidence, and risk were generated in one pass;
3. **Explainability** — attention weights and feature importance made forecasts easier to interpret;
4. **Multi-horizon support** — the same model produced forecasts from one month to twenty-four months ahead;
5. **Operational integration** — the model worked within a live system fed by external APIs rather than static files alone.

### 4.8.3 Reliability of Forecasts

Reliability was improved through confidence scores, risk levels, and data lineage metadata. Users can see not only the forecast value but also how confident the model is and which data sources supplied the input indicators. The API-first selection policy further increased reliability by reducing dependence on outdated manual entries.

### 4.8.4 Importance of External Economic Indicators

The results confirm that inflation prediction cannot depend on CPI alone. Exchange rate, interest rate, oil prices, money supply, and event-based features all contributed to model behaviour. This supports the design decision to integrate multiple APIs and to preserve multivariate input rather than reducing the problem to a univariate time series.

### 4.8.5 Unexpected Observations

Two observations deserve attention.

First, some country-level R² values were negative on very small test samples. This does not necessarily mean the model failed completely; it often happens when the test set is too small or when actual inflation moved within a narrow range while predictions varied slightly. The overall R² of 0.587 across all backtest records is the more meaningful figure.

Second, performance for Nigeria was weaker during sharp inflation spikes than during moderate periods. This is consistent with literature on emerging-market inflation, where exchange rate pass-through and policy shocks can create sudden jumps that are hard to forecast with monthly macro data alone.

### 4.8.6 Comparison with Existing Literature

Traditional ARIMA models often perform well on stable series but struggle with multivariate macroeconomic interactions. Deep learning models such as LSTM networks improve nonlinear modelling but may still lose long-range dependencies. Transformer-based approaches have increasingly been reported as competitive in time-series forecasting because attention can link distant time steps directly.

The results obtained in this project are broadly in line with that trend. The system did not outperform every reported benchmark in the literature, especially for long horizons and sparse country datasets, but it demonstrated that a TS-Transformer can be practically deployed in a real application with acceptable error levels and useful explainability.

### 4.8.7 Practical Applications

The developed system has several practical uses:

- **Economic forecasting** — institutions and analysts can obtain short- and medium-term inflation projections;
- **Policy formulation** — policymakers can use forecasts and scenario tools to examine possible outcomes under changing interest rates or spending levels;
- **Financial planning** — businesses exposed to import costs or currency risk can monitor inflation and exchange rate forecasts together;
- **Research** — academics can use the explainability and backtesting modules to study model behaviour;
- **Investment analysis** — investors can combine news sentiment, macro indicators, and forecast trends when assessing country risk.

## 5.2 Summary of the Study

Inflation and deflation remain central concerns for households, businesses, and policymakers, particularly in economies where price movements are influenced by exchange rate volatility, energy prices, and fiscal policy. Existing forecasting approaches often rely on univariate statistical models or on machine learning methods that are difficult to deploy in everyday decision-making because they lack interpretability and convenient access.

This study set out to design and implement a web-based platform that could forecast inflation and deflation risk using a TS-Transformer with multi-head self-attention. The platform, implemented as Velora, combines a Next.js frontend, a FastAPI backend, a PostgreSQL database, and a PyTorch forecasting engine. Economic indicators are sourced through an API-first selection layer that draws from FRED, IMF, World Bank, Trading Economics, and Exchange Rate API, with manual values used only when no API provides the required field.

Beyond raw prediction, the system was extended with intelligence features such as explainability panels, backtesting, scenario analysis, report generation, notifications, and role-based dashboards for users, analysts, and administrators. The model was trained on multivariate macroeconomic sequences for Nigeria, the United States, the United Kingdom, Ghana, and South Africa, and evaluated through walk-forward backtesting on held-out test windows.

The implementation phase confirmed that the proposed architecture is workable in practice. 4. The API-first indicator selection policy improved data freshness and traceability, although API coverage remains limited for some countries until more external sources are fully synchronised.
5. Multi-horizon forecasting (1, 3, 6, 12, and 24 months) was implemented successfully, with shorter horizons generally showing lower error than longer horizons.
6. The Intelligence Hub and related services demonstrated that a forecasting model becomes more useful when embedded in dashboards, warnings, lineage tracking, and narrative summaries rather than presented as isolated numbers.

## 5.4 Conclusion

The study achieved its main objective of developing a functional inflation and deflation prediction system based on the TS-Transformer with attention mechanism. The system was not limited to offline experimentation; it was implemented as a complete web application capable of ingesting live economic data, generating forecasts, storing results, and presenting them through role-specific interfaces.

The evaluation results support the conclusion that transformer-based time-series models can be applied effectively to macroeconomic forecasting when combined with careful preprocessing, multivariate feature design, and attention-driven explainability. An overall accuracy of approximately 80% is a credible outcome for inflation prediction, especially given the nonlinear and policy-sensitive nature of price movements in emerging markets such as Nigeria.

The project also showed that software engineering decisions matter as much as model choice. API integration, single-source indicator selection, database persistence, authentication, and reporting were all necessary to transform a research model into a practical decision-support tool. In that sense, the work contributes both to applied machine learning and to the design of intelligent economic information systems.

## 5.5 Recommendations

Based on the implementation experience and evaluation outcomes, the following recommendations are offered:

1. **Policymakers and central banks** should consider AI-assisted forecasting platforms as supplementary tools for monitoring inflation risk, especially when such platforms provide explainability and data lineage rather than black-box outputs alone.
2. **Researchers and students** working on economic forecasting should prioritise multivariate models and attention-based architectures when the goal is to capture long-range dependencies across macroeconomic variables.
3. **Developers of similar systems** should adopt an API-first data strategy so that forecasts are grounded in current official and market releases instead of stale manual tables.
4. **Institutions deploying the platform** should begin with countries that have richer historical records and stronger API coverage, then expand gradually as more data sources are connected.
5. **Analysts using the system** should interpret forecasts together with confidence scores, risk levels, and news sentiment rather than relying on point estimates alone.
6. **Administrators maintaining the platform** should monitor API sync health, retrain the model periodically as new macroeconomic data become available, and review backtest performance after each major model update.

## 5.6 Limitations of the Study

Despite the positive outcomes, several limitations should be acknowledged.

First, the size of the test set varied considerably across countries. Nigeria contributed most of the evaluation records, while the United States, United Kingdom, Ghana, and South Africa each contributed only five records in the current backtest split. Country-level metrics for those smaller samples should therefore be interpreted cautiously.

Second, macroeconomic forecasting remains sensitive to structural breaks. Sudden exchange rate adjustments, subsidy removals, conflict-related supply shocks, and global commodity spikes can produce inflation movements that historical monthly data alone may not anticipate.

Third, API coverage is not yet uniform across all 251 countries supported in the platform catalogue. For some countries, the system still depends on manual or locally stored fallback values, which reduces the benefit of the API-first policy until synchronisation improves.

Fourth, the current evaluation focused primarily on one-step-ahead inflation accuracy in the backtest module. Longer horizons and auxiliary outputs such as deflation probability and trend classification were implemented but were not evaluated with the same depth as the main inflation regression head.

Fifth, training and heavy experimentation were conducted on a standard development machine without large-scale GPU clusters. This was sufficient for the project scope but may limit the speed and scale of future experimentation.

## 5.7 Suggestions for Future Work

Several extensions could strengthen the platform and deepen the research contribution:

1. **Expand training data** by incorporating additional countries, higher-frequency series, and sectoral price indices such as food, energy, and core inflation components.
2. **Improve API coverage** by adding national statistical office connectors for more African economies, including direct NBS and CBN automation beyond manual imports.
3. **Introduce ensemble or hybrid models** that combine the TS-Transformer with econometric baselines such as VAR or dynamic factor models for comparison and possible accuracy gains.
4. **Add probabilistic forecasting** through quantile regression or conformal prediction so that users receive calibrated uncertainty intervals rather than only scalar confidence scores.
5. **Develop automated retraining pipelines** that schedule model updates when new macroeconomic releases are detected, with approval workflows for administrators.
6. **Conduct a formal user study** with economists, analysts, and students to measure whether explainability features actually improve trust and decision quality.
7. **Benchmark against published models** using standard inflation forecasting competitions and publicly available datasets to position the system more precisely within the literature.
8. **Deploy a production cloud instance** with monitoring, logging, and failover so that the platform can be tested under real institutional usage rather than localhost development alone.

This project designed, implemented, and evaluated an inflation and deflation prediction system built around the TS-Transformer with attention mechanism. The resulting platform integrates data ingestion, machine learning inference, explainability, reporting, and multi-role dashboards into one coherent application. Backtesting produced an overall forecast accuracy of 80.1%, demonstrating that the approach is feasible for practical economic monitoring.

The study concludes that attention-based transformers are a promising tool for macroeconomic forecasting when supported by sound software architecture and transparent data handling. With expanded data coverage, stronger API integration, and continued model refinement, the system can serve as a useful foundation for research, policy support, and financial decision-making in Nigeria and other monitored economies.
